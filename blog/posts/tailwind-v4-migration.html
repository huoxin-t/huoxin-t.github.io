Tailwind CSS v4 迁移实战指南

2025-08-10 · 阅读约 7 分钟

Tailwind CSS v4 是一次重大更新，核心变化是CSS-first 配置——告别 tailwind.config.js 和 PostCSS，改用原生 CSS 管理一切。本文记录从 v3 迁移到 v4 的完整过程。

核心变化一览

方面 v3 v4
配置方式 JS 文件 (tailwind.config.js) CSS (@theme 块)
构建管道 PostCSS 插件 Vite / Webpack 插件
内容扫描 content 配置项 自动扫描（基于 Vite 模块图）
CSS 入口 @tailwind 指令 @import 'tailwindcss'
自定义主题 extend.theme @theme { --color-* }

第一步：安装与插件配置

卸载 v3 相关依赖，安装 v4 包： 

npm uninstall tailwindcss postcss autoprefixer
npm install -D tailwindcss @tailwindcss/vite

在 vite.config.js 中注册插件，替代原来的 PostCSS： 

import tailwindcss from '@tailwindcss/vite'

export default defineConfig({
  plugins: [tailwindcss()],
})

可以删除 postcss.config.js 了。

第二步：迁移 CSS 入口

将原来的三条指令：

/* v3 — 删除 */
@tailwind base;
@tailwind components;
@tailwind utilities;

替换为一行：

/* v4 — 一行搞定 */
@import 'tailwindcss';

第三步：迁移主题配置

这是工作量最大的部分。v3 的 tailwind.config.js 中的主题扩展需要迁移到 CSS 的 @theme 块： 

/* v3 tailwind.config.js */
theme: {
  extend: {
    colors: {
      brand: { 500: '#4c6ef5', 600: '#4263eb' }
    },
    fontFamily: {
      sans: ['Inter', 'system-ui', 'sans-serif']
    }
  }
}

迁移为：

/* v4 style.css */
@theme {
  --color-brand-500: #4c6ef5;
  --color-brand-600: #4263eb;
  --font-sans: 'Inter', system-ui, sans-serif;
}
命名规则：v3 的 colors.brand.500 变成 --color-brand-500。 类名使用方式不变：bg-brand-500。

第四步：处理废弃的类名

v4 移除或重命名了一些工具类，需要注意全局替换：

  • flex-shrink-0 → 替换为 shrink-0
  • flex-grow-0 → 替换为 grow-0
  • decoration-slice → 已移除，使用默认行为
  • placeholder-* 系列 → 部分重命名

建议在项目中全局搜索 flex-shrink、 flex-grow、 decoration- 并做替换。

迁移后的收益

  • 构建速度提升约 30% — 不再需要 PostCSS 处理管道。
  • CSS 产物体积减小 — 自动内容扫描更精准，未使用的类不再生成。
  • 配置更直观 — 在 CSS 中定义设计令牌，与 W3C 自定义属性规范一致。
  • 编辑器体验改善 — @theme 中的定义可以直接被 CSS 语言服务器识别。

踩坑记录

  1. @config 指令兼容：v4 支持 @config "./tailwind.config.js" 加载旧配置，但部分 v3 特性（如 safelist）不生效。 建议彻底迁移到 @theme。
  2. 第三方插件兼容性：许多 v3 插件尚未适配 v4。 迁移前检查关键插件（如 typography、forms）的 v4 兼容状态。
  3. CSS 文件顺序：v4 中 @import 'tailwindcss' 必须在其它自定义样式之前，否则层叠优先级可能出错。

总结

Tailwind CSS v4 的迁移工作量主要集中在前三步：安装插件、改 CSS 入口、迁移 @theme 配置。 对于中小型项目（< 50 个组件），整个迁移过程大约需要 1–2 小时。 迁移后的构建速度和开发体验提升是值得的——尤其对于 Vite 用户，v4 的集成更加原生、简洁。
