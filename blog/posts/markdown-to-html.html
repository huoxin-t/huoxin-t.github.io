静态站点中渲染 Markdown 的几种方案

2025-11-20 · 阅读约 6 分钟

Markdown 是技术写作者最常用的格式。但在静态站点中如何将 .md 文件变成可访问的 HTML 页面？本文对比三种主流方案，分析各自的适用场景。

方案一：客户端运行时渲染

使用 marked、markdown-it 等库在浏览器中解析并渲染 Markdown： 

import { marked } from 'marked'

const res = await fetch('/posts/hello.md')
const md   = await res.text()
document.getElementById('content').innerHTML = marked.parse(md)
  • 优点：无需构建，写完 .md 直接部署；GitHub Pages 友好。
  • 缺点：首屏白屏时间增加；SEO 不友好（搜索引擎看到空 div）；用户需要下载 JS 解析器（marked ~50KB gzipped）。
  • 适用场景：文章数 < 10 篇的个人笔记站、原型阶段。

方案二：构建时转换

在 Vite/Webpack 构建阶段将 Markdown 预处理为 HTML，产出的就是纯静态页面。以 Vite 为例： 

// vite.config.js
import { plugin as mdPlugin, Mode } from 'vite-plugin-markdown'

export default {
  plugins: [mdPlugin({ mode: [Mode.HTML] })]
}

// 在组件中直接导入 Markdown
import { html } from '@/posts/hello.md'
document.getElementById('content').innerHTML = html
  • 优点：输出纯 HTML，首屏快、SEO 完美；可配合代码高亮、frontmatter 等插件。
  • 缺点：新增文章需要重新构建；Vite 插件生态中 Markdown 方案还不够成熟。
  • 适用场景：文章数 10–100 篇的个人博客或文档站。

方案三：直接写 HTML（推荐起步）

放弃 Markdown，直接用 HTML + Tailwind 类名写文章。就是本博客当前的做法：

  • 优点：零依赖、零运行时开销；完全掌控样式；构建速度最快。
  • 缺点：HTML 标签比 Markdown 啰嗦；迁移到其他平台时需要转换格式。
  • 适用场景：所有阶段的个人博客，尤其是注重性能和简洁性的站点。
我的建议：初期用方案三（HTML），文章数超过 20 篇后迁移到方案二（构建时转换）。 方案一（客户端渲染）仅适合完全不使用构建工具的场景。

对比总结

维度 客户端渲染 构建时转换 直接 HTML
首屏性能 差 优 最优
SEO 差 优 优
编写体验 优 优 一般
部署复杂度 简单 一般 简单
可扩展性 差 优 一般

以上对比基于 2025 年的工具生态，各项评分可能随时间改变。
